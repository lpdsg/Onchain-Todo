## 3. 本周遇到的问题和解决过程
问题 1：不确定合约是否部署成功

合约部署后，我最开始无法判断 Remix 中显示的内容是否代表部署成功，也不确定哪个字符串是完整合约地址。

解决方法：

查看 Remix 的 Deployed Contracts 区域；
区分合约地址和交易 Hash；
将合约地址复制到区块浏览器查询；
核对部署交易状态。

最终确认合约已经成功部署到 Monad Testnet。

- 问题 2：无法区分合约地址和交易 Hash

刚开始看到多个以 0x 开头的字符串时，我不清楚它们分别代表什么。

后来确认：

合约地址通常是 42 个字符；
交易 Hash 通常是 66 个字符；
合约地址代表部署后的合约账户；
交易 Hash 代表某一次具体链上交易。
- 问题 3：不理解读取函数和写入函数的区别

最开始我不清楚为什么调用 createTodo 需要钱包确认，而调用 getMyTodos 不需要。

后来理解：

createTodo 修改合约状态，因此需要发起交易；
getMyTodos 只读取已有数据，不修改状态；
写入操作需要消耗 Gas；
读取操作通常可以本地调用。
- 问题 4：不知道 AI 生成代码是否可以直接使用

我意识到 AI 可以快速生成 Solidity 代码，但生成结果不能直接默认安全和正确。

因此，我对以下内容进行了人工检查：

Solidity 版本是否兼容；
是否使用 msg.sender 隔离不同用户；
数组下标是否有越界检查；
写入函数是否正确修改状态；
是否包含不必要的管理员权限；
是否加入 Token、转账或其他高风险功能；
是否存在私钥、助记词或 API Key 泄露风险。

## 6. 我对 Monad 和 Web3 的理解变化

在 Week 1 开始前，我对 Web3 的理解主要停留在钱包、Token 和区块链概念层面。

完成实际操作后，我的理解发生了以下变化：

### 6.1 钱包不仅是资产工具

钱包也是 Web3 应用中的用户账户和身份入口。

在 DApp 中，用户通常通过钱包地址识别身份，并使用私钥签名证明操作是由本人发起的。

### 6.2 智能合约是公开运行的程序

智能合约不是普通服务器上的后端代码，而是部署到区块链网络上的程序。

部署后：

合约拥有固定地址；
用户可以调用合约函数；
写入结果记录在链上；
交易可以通过区块浏览器验证。
### 6.3 上链并不代表所有内容都必须放在链上

链上记录适合需要公开、可验证和难以随意修改的数据。

但头像、页面样式、普通文本展示和复杂排行榜计算等内容，不一定全部需要上链。

合理的产品应该考虑链上和链下的分工。

### 6.4 Monad 的价值不能只概括为“更快”

对于签到、游戏、任务系统和排行榜等高频交互产品，用户会频繁发起交易。

如果交易确认慢或 Gas 成本高，用户体验会明显下降。

Monad 的高性能、低延迟和 EVM 兼容性，可能帮助开发者在保留以太坊开发工具和合约生态的同时，构建更接近普通互联网体验的链上应用。

- 收获 2：理解了 AI 不能代替人工验证

AI 可以生成代码和解释步骤，但以下内容必须由开发者自己检查：

代码逻辑；
合约权限；
钱包网络；
交易签名；
链上结果；
敏感信息安全。

- 收获 3：开始理解链上与链下的边界

不是所有数据都应该写入区块链。

真正需要上链的通常是：

需要公开验证的状态；
与奖励、积分或资格相关的数据；
不希望被项目方随意修改的记录。

其他展示型或高成本数据可以保留在链下。
